# Invitation Sites — архитектура: Next.js-каталог + Astro-приглашения

**Статус:** принятая целевая граница. Переход выполняется поэтапно, без замены работающего Astro-сайта.

## 1. Зачем разделение

Каталог — развивающаяся бизнес-витрина: категории, подборки, статьи, страницы услуг, SEO-лендинги, лид-магниты и будущая owner-админка. Для него выбирается **Next.js** как отдельный frontend.

Каждое приглашение — лёгкая самостоятельная публичная страница с собственной визуальной историей. Для них остаётся **Astro**, чтобы не утяжелять клиентские сайты.

## 2. Карта системы

```text
Посетитель
   │
   ├─ каталог, категории, статьи, услуги, будущая админка
   │  └─ Next.js: catalog-next/
   │
   ├─ живое демо выбранного дизайна
   │  └─ Astro: astro-site/src/pages/templates/[slug].astro
   │
   └─ бриф и заявка
      └─ Astro: astro-site/src/pages/brief.astro
          └─ Supabase Edge Function → requests

Владелец
   └─ защищённая работа с заявками
      └─ Astro admin + owner-requests Edge Function
```

## 3. Границы ответственности

| Слой | Технология | Ответственность |
|---|---|---|
| Каталог | Next.js | главная, фильтры, категории, будущие SEO-страницы, услуги, контент и вход в воронку |
| Приглашения | Astro | страницы `/templates/<slug>/`, визуальная анимация события, данные конкретного клиента |
| Бриф | Astro | выбор шаблона, сбор данных, отправка заявки |
| Заявки | Supabase Edge Function | безопасный приём и owner-only чтение данных |
| Данные шаблонов | временно Astro `templates.ts` | до выделения общего пакета; Next-каталог получает только публичную витринную проекцию |

## 4. Пользовательский путь

```text
Next.js: каталог
  → Astro: живое демо /templates/<slug>/
  → Astro: /brief/?template=<slug>
  → Edge Function: заявка
  → owner-зона: обработка заявки
```

Каталог не принимает заявки напрямую и не содержит service-role ключей. CTA «Выбрать» всегда сохраняет `slug` в query-параметре.

## 5. Адреса и развёртывание

На этапе разработки приложения запускаются отдельно. Перед публикацией требуется выбрать единый домен с прокси/rewrites:

- Next.js получает каталог и маркетинговые маршруты: `/`, `/catalog`, `/categories/*`, `/journal/*`, `/services/*`;
- Astro получает маршрутный namespace приглашений: `/templates/*`, `/brief`, а также временно `/admin/*`;
- Supabase остаётся серверным контуром.

До настройки домена Next-каталог не подменяет текущую опубликованную Astro-витрину.

## 6. Последовательность миграции

1. Создать отдельное приложение `catalog-next/` и проверить production build.
2. Перенести в Next.js чистый mobile-first каталог по утверждённой спецификации.
3. Подключить реальную публичную проекцию готовых шаблонов и ссылки на Astro-демо/бриф.
4. Проверить путь `каталог → демо → бриф → заявка` локально и на целевом домене.
5. Только после этого убрать прежнюю Astro-главную из публичного маршрута.
6. Отдельно расширять Next.js страницами категорий, услуг, кейсов, журнала и owner-функциями.

## 7. Правила безопасности и совместимости

- Не менять существующие Astro-маршруты приглашений во время создания Next-каталога.
- Не переносить в Next.js service-role credentials или owner token.
- Использовать только данные готовых шаблонов; будущие категории без шаблонов не показывать.
- Сначала сохранять параллельный запуск и реальную сборку, затем менять публичную маршрутизацию.
