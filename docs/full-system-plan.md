# Полный план системы: платформа сайтов-приглашений

Этот документ — главный план проекта. По нему строим не просто MVP, а готовую систему, которую можно показывать клиентам, продавать и масштабировать.

## 1. Главная идея

Мы делаем **мини-фабрику цифровых приглашений**.

Клиент не должен просто писать в WhatsApp и ждать непонятно чего. У него должен быть понятный путь:

1. Открывает сайт.
2. Смотрит каталог шаблонов.
3. Фильтрует по типу мероприятия: свадьба, никах, кыз узатуу, день рождения, baby shower, юбилей, тушоо той.
4. Открывает демо шаблона.
5. Нажимает «Заказать такой сайт».
6. Заполняет бриф.
7. Отправляет фото, дату, место, текст, музыку, пожелания.
8. Владелец видит заказ в админке.
9. Система/агенты помогают проверить данные, собрать сайт, подготовить Instagram-контент и опубликовать ссылку.
10. Клиент получает готовую публичную ссылку.
11. Гости открывают приглашение и отправляют RSVP/ответ.

Главная цель: **сделать готовый бизнес-процесс, а не одну красивую страницу**.

## 2. Из каких частей состоит система

### 2.1 Публичная витрина

Это сайт, который видят клиенты.

Нужные страницы:

- Главная страница.
- Каталог шаблонов.
- Страница одного шаблона.
- Демо приглашения.
- Страница цен/пакетов.
- Страница «Как заказать».
- Страница брифа/анкеты.
- FAQ.
- Контакты: WhatsApp, Instagram.

Главная задача витрины — убедить клиента, что это удобно, красиво и быстро.

---

### 2.2 Каталог шаблонов

Каталог обязателен. Клиент должен выбирать сам, как в магазине.

У каждого шаблона должны быть:

- название;
- тип мероприятия;
- стиль: luxury, minimal, romantic, traditional, modern, kids, pastel;
- цена или пакет;
- демо-ссылка;
- превью-картинка;
- список секций;
- доступные цвета;
- поддержка языков: русский, кыргызский, казахский, английский;
- кнопка «Посмотреть демо»;
- кнопка «Заказать этот шаблон».

Фильтры каталога:

- тип мероприятия;
- цена;
- стиль;
- язык;
- наличие RSVP;
- наличие музыки;
- наличие фото/галереи;
- premium/simple.

Сначала делаем 5–10 сильных шаблонов, потом расширяем каталог.

---

### 2.3 Бриф / форма заказа

После выбора шаблона клиент заполняет форму.

Форма должна собирать:

- тип мероприятия;
- выбранный шаблон;
- имена;
- дата;
- время;
- место;
- адрес;
- ссылка на 2GIS/Google Maps;
- программа мероприятия;
- dress code;
- текст приглашения;
- фото;
- музыка;
- язык сайта;
- WhatsApp/Instagram клиента;
- дедлайн;
- пожелания по цвету/стилю;
- нужна ли RSVP форма;
- нужен ли обратный отсчёт;
- нужна ли галерея;
- нужен ли отдельный текст для родителей/семьи.

Форма не должна быть сложной. Если клиент не знает, что писать, система должна помогать примерами.

---

### 2.4 Админка владельца

Это панель, через которую владелец управляет бизнесом.

В админке нужны разделы:

1. **Заявки**
   - новые;
   - в работе;
   - ждём данные;
   - ждём оплату;
   - на проверке клиента;
   - опубликовано;
   - завершено.

2. **Клиенты**
   - имя;
   - контакты;
   - история заказов;
   - заметки.

3. **Сайты клиентов**
   - slug/ссылка;
   - выбранный шаблон;
   - данные мероприятия;
   - статус публикации;
   - дата создания;
   - дата мероприятия.

4. **Шаблоны**
   - список шаблонов;
   - активен/скрыт;
   - цена;
   - категории;
   - превью;
   - демо.

5. **RSVP / ответы гостей**
   - имя гостя;
   - придёт/не придёт;
   - количество людей;
   - комментарий;
   - дата ответа.

6. **Instagram/контент**
   - идеи постов;
   - готовые captions;
   - stories;
   - reels scripts;
   - контент-календарь;
   - статусы публикаций.

7. **Задачи агентов**
   - что нужно проверить;
   - что нужно сгенерировать;
   - что ждёт владельца;
   - что уже готово.

### 2.5 База данных и хранение

Для готовой системы используем **Supabase**:

- Postgres — таблицы и данные;
- Supabase Storage — фото, музыка, превью;
- Auth — вход владельца в админку;
- Row Level Security — защита данных.

GitHub Pages можно оставить для демо. Но полноценная система с заказами, админкой и базой лучше на:

- Vercel или Netlify для сайта;

## 3. Как владелец будет управлять системой

Владелец не должен лазить по коду для каждого заказа.

Нормальный процесс:

1. Владелец открывает админку.
2. Видит новые заявки.
3. Открывает заявку.
4. Проверяет данные клиента.
5. Если чего-то не хватает — система показывает список недостающих данных.
6. Владелец отправляет клиенту готовый WhatsApp-текст: «Нужно дослать фото/адрес/музыку».
7. Когда данные готовы, владелец нажимает «Создать сайт».
8. Система создаёт страницу на основе шаблона и данных.
9. Владелец смотрит preview.
10. Если всё нормально — нажимает «Опубликовать».
11. Клиент получает ссылку.
12. RSVP гостей собирается в админке.

Владелец должен видеть не просто файлы, а понятный статус каждого заказа.

## 4. AI/агенты внутри проекта

Агенты — это не отдельная магия. Это роли/модули, которые помогают владельцу быстрее делать работу.

### 4.1 Owner/Admin Agent

Главный помощник владельца.

Что делает:

- показывает, какие заказы срочные;
- говорит, что нужно сделать сегодня;
- проверяет, где не хватает данных;
- предлагает следующий шаг;
- готовит текст клиенту;
- помогает не забыть дедлайны.

Пример:

> «У заказа A нет карты и фото. У заказа B всё готово, можно публиковать. Сегодня нужно подготовить 2 stories и проверить RSVP по свадьбе 20 июля.»

---

### 4.2 Order/Brief Agent

Агент обработки заявок.

Что делает:

- читает заполненный бриф;
- проверяет недостающие поля;
- нормализует дату, время, место;
- предлагает красивый текст приглашения;
- готовит список вопросов клиенту;
- переводит текст на нужный язык;
- подготавливает `event data` для генератора сайта.

---

### 4.3 Template/Design Agent

Агент шаблонов.

Что делает:

- предлагает новые шаблоны;
- улучшает существующие секции;
- ищет legal/free/purchased/user-provided assets;
- проверяет, чтобы шаблон был mobile-first;
- готовит структуру секций;
- помогает делать варианты под свадьбу, никах, кыз узатуу, birthday.

Правило: не копировать чужие сайты 1-в-1. Берём структуру/уровень/ощущение, но делаем оригинальный шаблон.

---

### 4.4 QA/Deploy Agent

Агент проверки и публикации.

Что делает:

- запускает build;
- проверяет Astro check;
- открывает сайт в браузере;
- проверяет mobile view;
- проверяет console errors;
- проверяет ссылки, карты, фото, RSVP;
- готовит отчёт: что работает, что сломано;
- помогает деплоить.

---

### 4.5 Instagram/Content Agent

Агент Instagram.

Что делает:

- создаёт контент-план;
- пишет captions;
- готовит stories;
- предлагает reels scripts;
- делает идеи для каруселей;
- готовит ответы клиентам в Direct;
- превращает готовые шаблоны в посты: «Новый шаблон Wedding Luxury»;
- готовит еженедельный план продвижения.

Важно: агент может готовить контент и тексты, но публикацию лучше сначала делать вручную владельцем. Полная автоматическая публикация Instagram подключается позже, когда будет безопасный официальный API/инструмент.

## 5. End-to-end workflow заказа

Полный путь одного заказа:

```text
Instagram/сайт
  → каталог шаблонов
  → демо шаблона
  → бриф клиента
  → запись в Supabase
  → админка владельца
  → проверка данных Order Agent
  → выбор/подтверждение шаблона
  → генерация client site
  → preview
  → правки
  → оплата
  → публикация
  → отправка ссылки клиенту
  → RSVP гостей
  → финальный отчёт/архив
```

## 6. Технический стек

### Сейчас

Уже есть Astro-проект и GitHub Pages demo.

Используем это для:

- публичных демо шаблонов;
- каталога;
- быстрых визуальных проверок;
- GitHub Pages публикации.

### Готовая версия

Рекомендуемый стек:

- **Astro** — публичная витрина, каталог, шаблоны, быстрые страницы.
- **Supabase** — база, auth, storage, RSVP, заказы.
- **Vercel/Netlify** — продакшен-деплой с backend/API routes.
- **GitHub** — код, история, шаблоны, деплой.
- **WhatsApp links** — быстрый контакт с клиентом.
- **Instagram** — продажи и контент.

Если позже потребуется сложная админка и много интерактива, можно добавить Next.js/React dashboard. Но начинать можно с Astro + Supabase.

## 9. Генерация сайта клиента

Каждый сайт клиента строится из трёх вещей:

```text
template + event data + assets = public invitation site
```

Пример:

```text
wedding-luxury-01
+ Асан & Айсулуу, дата, ресторан, программа, фото
+ uploaded images/music
= /invites/asan-aisuluu/
```

Сначала генерация может быть простой:

- данные лежат в JSON/TS файле;
- Astro создаёт страницу по slug;
- публикация через build/deploy.

Потом делаем полноценный вариант:

- данные из Supabase;
- файлы из Supabase Storage;
- сайт рендерится по slug;
- владелец меняет данные в админке без кода.

## 11. Instagram workflow

Instagram нужен не просто как соцсеть, а как канал продаж.

### Что должно быть в проекте

- контент-план;
- шаблоны постов;
- шаблоны stories;
- тексты для Direct;
- highlights: Templates, Prices, Reviews, How to order, FAQ;
- weekly content calendar;
- список идей reels.

### Процесс

1. Сделали новый шаблон.
2. Template Agent готовит описание преимуществ.
3. Instagram Agent делает:
   - caption;
   - story text;
   - reels idea;
   - CTA в WhatsApp.
4. Владелец проверяет.
5. Публикует вручную.
6. Все заявки идут в бриф/админку.

## 15. Короткий итог

Мы строим систему из 7 главных блоков:

1. Публичная витрина.
2. Каталог шаблонов.
3. Бриф/заказ.
4. База данных и файлы.
5. Админка владельца.
6. Генератор сайтов клиентов.
7. Instagram + AI/операционные агенты.

Только после этого дизайн-шаблоны становятся частью нормального бизнеса, а не отдельными страницами вручную.
