# Owner workflow: как владелец управляет заказами

Этот документ описывает рабочий процесс владельца платформы сайтов-приглашений: от первого сообщения клиента в Instagram/WhatsApp до опубликованной ссылки, RSVP и закрытия заказа.

## 1. Главная идея

Владелец не должен каждый раз держать весь процесс в голове. У каждого заказа должен быть понятный путь:

```text
Сообщение клиента
  → выбор пакета и шаблона
  → сбор данных через бриф
  → проверка данных
  → создание заказа в системе
  → генерация сайта
  → preview
  → правки клиента
  → публикация
  → отправка ссылки
  → RSVP/ответы гостей
  → закрытие заказа и отзыв
```

Главная задача owner workflow — сделать так, чтобы заказ не терялся и всегда было понятно:

- кто клиент;
- какой шаблон выбран;
- какие данные уже есть;
- чего не хватает;
- на каком статусе заказ;
- что нужно сделать дальше;
- кто отвечает за следующий шаг.

---

## 2. Роль владельца/админа

Владелец платформы отвечает за:

1. Приём заявок.
2. Общение с клиентом.
3. Объяснение пакетов и цен.
4. Подбор шаблона.
5. Проверку брифа.
6. Загрузку фото/музыки/адреса/текста.
7. Создание или запуск генерации сайта.
8. Проверку preview.
9. Внесение правок.
10. Публикацию ссылки.
11. Отправку сайта клиенту.
12. Контроль RSVP.
13. Закрытие заказа и сбор отзыва.

В полной версии часть этих действий будут помогать делать AI/операционные агенты.

---

## 3. Каналы входа заявок

### 3.1 Instagram

Самый важный канал продаж.

Клиент может написать:

- в Direct;
- под постом;
- после просмотра Reels;
- после Story;
- после рекламы.

Что делает владелец:

1. Быстро отвечает клиенту.
2. Спрашивает тип мероприятия.
3. Отправляет ссылку на каталог шаблонов.
4. Уточняет дату мероприятия.
5. Предлагает пакет.
6. Отправляет форму брифа.

Пример короткого ответа:

```text
Здравствуйте! Да, мы делаем сайты-приглашения.
Напишите, пожалуйста:
1. Какое мероприятие?
2. Какая дата?
3. Хотите простой, стандартный или premium вариант?

Каталог шаблонов: [link]
```

### 3.2 WhatsApp

WhatsApp нужен для быстрого общения, подтверждений и отправки ссылки.

Что делает владелец:

- отправляет клиенту бриф;
- получает фото/музыку;
- уточняет недостающие данные;
- отправляет preview;
- отправляет финальную ссылку.

### 3.3 Сайт/каталог

Клиент может сам открыть каталог и нажать:

```text
Заказать этот шаблон
```

После этого он попадает на форму заказа.

## 4. Полный путь заказа

## Шаг 1. Клиент пишет в Instagram/WhatsApp

Цель: понять, что нужно клиенту.

Владелец уточняет:

- тип мероприятия;
- дату;
- город/страну;
- примерный стиль;
- нужен ли RSVP;
- нужен ли premium дизайн;
- срочность.

Статус заказа:

```text
new
```

---

## Шаг 2. Владелец предлагает пакет и шаблон

Пакеты могут быть такими:

| Пакет | Для кого | Что входит |
|---|---|---|
| Basic | простой сайт | шаблон, дата, место, текст, ссылка |
| Standard | нормальный заказ | шаблон, фото, музыка, карта, RSVP |
| Premium | дорогой вариант | уникальный стиль, анимации, несколько секций, wow-эффект |

Владелец отправляет клиенту:

- 2–5 подходящих шаблонов;
- цену;
- что входит;
- срок выполнения.

Статус:

```text
new
```

или

```text
brief_received
```

если клиент уже готов заполнить данные.

---

## Шаг 3. Клиент заполняет бриф

Бриф собирает данные:

- имя клиента;
- WhatsApp/Instagram;
- тип мероприятия;
- имена главных людей;
- дата;
- время;
- место;
- адрес;
- карта/2GIS;
- программа;
- dress code;
- текст приглашения;
- фото;
- музыка;
- язык;
- выбранный шаблон;
- пожелания.

Статус:

```text
brief_received
```

---

## Шаг 4. Владелец проверяет данные

Владелец открывает заказ и проверяет:

- все обязательные поля заполнены;
- дата правильная;
- время указано;
- адрес понятный;
- карта открывается;
- фото подходят по качеству;
- музыка загружена или выбрана;
- текст не содержит ошибок;
- выбранный шаблон подходит под мероприятие.

Если не хватает данных:

```text
missing_info
```

Если всё готово:

```text
ready_to_build
```

---

## Шаг 5. Владелец создаёт заказ в админке

В MVP это может быть вручную через Supabase Dashboard или локальные JSON/TS файлы.

В полной версии — через админку.

Что должно быть в заказе:

- customer;
- order;
- event;
- template;
- assets;
- статус;
- deadline;
- notes.

Статус:

```text
ready_to_build
```

---

## Шаг 6. Генерация сайта

Система берёт:

```text
template + event data + assets + settings
```

и создаёт клиентский сайт.

Пример public slug:

```text
/clients/asan-aisuluu/
```

или:

```text
/invites/asan-aisuluu/
```

Статус заказа:

```text
building
```

Статус сайта:

```text
draft
```

---

## Шаг 7. Preview

Когда сайт собран, владелец открывает preview и проверяет:

- hero section;
- имена;
- дату;
- время;
- адрес;
- карту;
- фото;
- музыку;
- countdown;
- RSVP форму;
- мобильную версию;
- скорость загрузки;
- нет ли ошибок в консоли.

Статус заказа:

```text
preview_ready
```

Статус сайта:

```text
preview
```

---

## Шаг 8. Клиент смотрит preview

Владелец отправляет клиенту preview-ссылку.

Клиент может ответить:

- всё хорошо;
- исправить текст;
- поменять фото;
- поменять музыку;
- изменить адрес;
- добавить программу;
- поменять цвет/стиль.

Если нужны правки:

```text
revision_needed
```

Если всё принято:

```text
client_review
```

или сразу:

```text
waiting_payment
```

---

## Шаг 9. Оплата

В зависимости от бизнес-правил:

- 100% предоплата;
- 50% до работы и 50% перед публикацией;
- оплата после preview;
- оплата вручную через перевод.

Статусы оплаты:

```text
unpaid
partial
paid
```

Если оплата нужна перед публикацией:

```text
waiting_payment
```

После оплаты:

```text
paid
```

---

## Шаг 10. Публикация сайта

После проверки и оплаты владелец публикует сайт.

Проверить перед публикацией:

- правильный slug;
- нет тестового текста;
- фото загружаются;
- музыка работает;
- RSVP отправляется;
- ссылка открывается с телефона;
- карта открывается;
- нет ошибок в браузере.

Статус заказа:

```text
published
```

Статус сайта:

```text
published
```

---

## Шаг 11. Отправка ссылки клиенту

Владелец отправляет клиенту:

```text
Ваш сайт-приглашение готов:
[link]

Проверьте, пожалуйста, что ссылка открывается у вас и у гостей.
```

Если включён RSVP, владелец объясняет:

```text
Гости смогут нажать кнопку и подтвердить участие.
Ответы будут сохраняться в системе.
```

---

## Шаг 12. RSVP и ответы гостей

После публикации гости могут отправлять ответы.

Владелец видит:

- имя гостя;
- придёт/не придёт;
- количество гостей;
- сообщение;
- дату ответа.

Дальше можно отправлять клиенту таблицу RSVP.

---

## Шаг 13. Закрытие заказа

Когда мероприятие прошло или работа завершена:

```text
completed
```

Владелец может:

- попросить отзыв;
- попросить разрешение показать сайт как кейс;
- сделать Instagram post/reel/story;
- сохранить шаблон как пример;
- отметить, что можно улучшить.

---

## 5. Статусы заказа

Текущий MVP использует один простой список статусов прямо в `/admin/requests/`. Этого достаточно, чтобы не терять заявки и понимать следующий шаг.

| Статус в системе | Значение | Что делает владелец |
|---|---|---|
| `brief_received` | новая заявка / бриф получен | открыть заявку и проверить данные |
| `contacted` | связались с клиентом | подтвердить детали и цену |
| `missing_info` | ждём данные | написать, чего не хватает |
| `ready_to_build` | готово к сборке | можно создавать сайт клиента |
| `in_progress` | в работе | собрать сайт и проверить данные |
| `preview_sent` | preview отправлен | ждать ответ клиента |
| `revision_requested` | нужны правки | внести изменения |
| `delivered` | сайт передан | отправить финальную ссылку |
| `paid` | оплачено | отметить заказ как оплаченный |
| `cancelled` | отменён | закрыть заявку |

## 6. Что владелец делает вручную в MVP

На первом этапе владелец вручную:

- отвечает в Instagram/WhatsApp;
- выбирает шаблон;
- отправляет форму брифа;
- проверяет данные;
- загружает фото;
- запускает создание страницы;
- проверяет сайт;
- отправляет ссылку клиенту;
- собирает правки;
- ведёт оплату;
- смотрит RSVP.

Это нормально для старта. Главное — чтобы процесс был понятный и повторяемый.

```text

[ ] Предложил пакет
[ ] Предложил шаблоны
[ ] Отправил бриф
```

После получения брифа:

```text
[ ] Проверил имена
[ ] Проверил дату
[ ] Проверил время
[ ] Проверил место
[ ] Проверил карту
[ ] Проверил текст
[ ] Проверил фото
[ ] Проверил музыку
[ ] Понял, чего не хватает
```

Перед preview:

```text
[ ] Сайт собран
[ ] Данные правильные
[ ] Фото открываются
[ ] Музыка работает
[ ] Карта работает
[ ] RSVP работает
```

После заказа:

```text
[ ] Заказ закрыт
[ ] Отзыв запрошен
[ ] Кейс можно использовать в Instagram

```

## 10. Где помогают AI/операционные агенты

### Owner/admin agent

Помогает владельцу видеть:

- что нужно сделать сейчас;
- какие заказы просрочены;
- где не хватает данных;
- какие клиенты ждут ответа.

### Order/brief agent

Проверяет бриф:

- заполнены ли обязательные поля;
- есть ли ошибки в дате/времени;
- достаточно ли данных для сайта;
- какой текст можно предложить.

### Template/design agent

Помогает:

- подобрать шаблон;
- предложить стиль;
- проверить визуальное качество;
- подготовить новый шаблон.

### QA/deploy agent

Проверяет сайт перед публикацией:

- build;
- ссылки;
- консоль браузера;
- mobile view;
- RSVP;
- изображения.

### Instagram/content agent

Готовит:

- post captions;
- stories;
- reels ideas;
- кейсы;
- продающие тексты.

## 11. MVP workflow

Минимальный рабочий процесс:

```text
Instagram/WhatsApp
  → клиент выбирает шаблон
  → клиент заполняет бриф
  → владелец вручную проверяет данные
  → данные добавляются в проект
  → сайт собирается
  → preview проверяется
  → ссылка отправляется клиенту
```

MVP можно вести даже без сложной админки, но все действия должны соответствовать будущей data model.

---

## 12. Full workflow

Полная версия:

```text
Каталог
  → форма заказа
  → Supabase
  → админка владельца
  → автоматическая проверка данных
  → генерация сайта
  → preview
  → оплата
  → публикация
  → RSVP dashboard
  → Instagram/CRM follow-up
```

## 13. Что строить дальше в коде

После этого документа правильный порядок разработки:

1. Создать data-driven структуру шаблонов в коде.
2. Сделать страницу заказа `/order/`.
3. Подключить выбранный template slug к форме заказа.
4. Сохранить заявку локально или в Supabase.
5. Создать простую owner/admin страницу со списком заказов.
6. Создать генератор клиентского сайта из данных заказа.
7. Добавить RSVP форму.
8. Добавить публикацию клиентской ссылки.

---

## 14. Итог

Owner workflow — это операционная карта бизнеса.

Если этот процесс работает, дальше можно масштабировать:

- больше шаблонов;
- больше типов мероприятий;
- автоматизация;
- AI-агенты;
- CRM;
- реклама;
- Instagram-контент;
- платные premium-пакеты.

Главное правило: каждый заказ должен иметь понятный статус, понятные данные и понятный следующий шаг.
